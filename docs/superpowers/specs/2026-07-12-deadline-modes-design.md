# 마감·시작시간 입력 개편 — 타임테이블 제거 후속 디자인 스펙

- 날짜: 2026-07-12
- 상태: 디자인 4개 섹션 사용자 승인됨, 스펙 검토 대기
- 작업 브랜치: `feature/deadline-modes` (dev에서 분기 — dev가 main 미머지 상태의 최신)

## 1. 배경과 목표

대시보드 개편(2026-07-11)으로 원형 타임테이블은 사라졌지만, 태스크 생성 폼과 시간 추론
파이프라인에는 "모든 태스크는 시간 슬롯"이던 시절의 잔재가 남아 있다:

1. 생성 폼의 마감 필드가 **오늘 23:59로 기본 채워져** 대부분의 태스크에 마감이 강제된다.
2. 안내 문구("비워두면 AI가 자동으로 채워줘요")가 실제 동작과 어긋난다 — AI 프롬프트는
   이미 "시점 단서 없으면 deadline null" 규칙(feature/ai-no-fake-deadline)을 따른다.
3. "이 일은 기한이 없다"를 **명시적으로 선언할 방법이 없다.** '언젠가' 분류는 AI 판별의
   부산물로만 발생한다.
4. 시작시간과 마감을 함께 입력하면 그 간격을 예상시간으로 파생(`Duration.between`)해,
   n박 일정(예: 시작 오늘·마감 3일 뒤)이 "예상 시간은 1분부터 1440분 사이" 검증에 걸려
   **등록 자체가 불가능**하다.

**목표**: 마감 입력을 명시적 4모드(AI 판별 / 오늘까지 / 언젠가 / 직접 입력)로 재편하고,
슬롯 시대의 파생 로직을 제거해 n박 일정과 기한 없는 일이 자연스럽게 공존하게 한다.

**설계 원칙**: *단서 없으면 지어내지 않는다.* AI는 제목·메모에 명시적/상대적 시점 단서가
있을 때만 마감·시작시간을 채운다. 예상시간(estimatedMinutes)은 시작~마감 "구간"이 아니라
**집중 작업량** 의미로 보존한다(1~1440분).

## 2. 생성 폼 (AddTaskModal) — 마감 모드 칩 4지선다

기존 마감 datetime 필드의 기본값(오늘 23:59)과 상단 안내 문구를 제거하고, 모드 칩 한 줄로
교체한다. 카테고리 칩과 같은 시각 패턴을 쓴다.

```
마감
[✨ AI가 알아서(기본)] [오늘까지] [🌙 언젠가] [📅 직접 입력]
```

| 모드 | 전송 값 | 동작 |
|---|---|---|
| ✨ AI가 알아서 (기본) | `deadline: null, noDeadline: false` | 백엔드가 제목·메모의 시점 단서로 판별 — 단서 있으면 마감 설정, 없으면 '언젠가' 생성 |
| 오늘까지 | `deadline: 오늘 23:59` | 기존 기본값 동작을 명시적 선택지로 이동 |
| 🌙 언젠가 | `deadline: null, noDeadline: true` | 마감 없음을 선언 — AI가 마감·시작시간을 추론하지 않음 |
| 📅 직접 입력 | `deadline: 사용자 입력값` | 선택 시에만 datetime 필드 등장(초기값 오늘 23:59에서 수정) |

- 칩 아래 한 줄 헬퍼 텍스트가 선택된 모드를 설명한다.
  예: AI 모드 → "시점 단서가 있으면 마감을 잡고, 없으면 '언젠가'로 분류해요"
- 시작 시간·예상 시간 체크박스 입력은 현행 유지. '언젠가' + 시작시간 직접 입력 조합도
  허용한다(막을 이유 없음 — 시작만 정한 기한 없는 일).
- 프론트 검증(마감은 미래·시작 이후)은 직접 입력 모드에서만 해당되며 현행 유지.

## 3. API·추론 파이프라인 (백엔드)

### 요청 필드

- `TaskRequest`·`TaskUpdateRequest`에 `noDeadline`(boolean, 기본 false) 추가.
- `noDeadline: true`이면 `inferScheduleIfNeeded`에서 마감·시작시간을 **추론으로 채우지 않는다**
  (마감은 null 확정, 시작시간은 유저가 직접 입력한 값만 보존).
  **예상시간 AI 추정은 유지** — 추천 문구("N분 정도라 지금 한 번 집중하기 좋아요")에 쓰이므로.
  구현은 추론 결과에서 예상시간만 취하고 마감·시작시간은 버리는 방식으로 단순화한다.

### 타임테이블 잔재 파생 3종 제거 (`TaskServiceImpl.inferScheduleIfNeeded`)

1. 시작+마감 → 예상시간 = `Duration.between(start, deadline)` 파생 제거 (n박 등록을 막던 범인)
2. 마감+예상시간 → 시작시간 = `deadline - estimatedMinutes` 역산 파생 제거
3. 부분 추론 경로의 동일 간격 파생 제거

파생이 사라진 자리는 "값 없음(null)"으로 둔다. 시작+마감이 모두 입력된 경우는 지금처럼
AI 호출 없이 통과하되 예상시간은 null 유지(유저가 원하면 체크박스로 직접 입력).

### `inferSchedule` AI 프롬프트 정비 (`OpenAiServiceImpl`)

- "startTime + estimatedMinutes → deadline", "deadline − estimatedMinutes → startTime",
  "start~deadline 간격 → estimatedMinutes" 등 **슬롯 산술 규칙 삭제**
- null 가드("단서 없으면 지어내지 않는다")를 **모든 입력 조합에 적용** — 현재는 세 필드가
  전부 빈 경우에만 걸려 있어, 예상시간만 입력하면 여전히 마감을 지어내는 구멍이 있음
- 시작시간도 동일 원칙: "내일 3시 회의"처럼 명시적 시간 단서가 있을 때만 채움
- 예상시간 추정 규칙(작업 유형 기반, 1~1440분)은 유지

### 유지하는 것

- `validateSchedule`: 1440분 캡, "마감은 시작 이후", 미래 마감 검증 모두 유지
- `endTime` 세팅 로직과 `effectiveEndTime` fallback(진행 중 판정) — 변경 없음
- AI 사용료: 태스크 생성 1점 현행 유지 — 언젠가 모드도 우선순위 스코어링은 돌기 때문

## 4. 수정 모달 (EditTaskModal)

- 생성 폼과 같은 모드 칩 UI를 적용하되, 초기 선택은 현재 값을 반영한다:
  마감 있음 → '직접 입력'(값 표시), 마감 없음 → '언젠가'
- '✨ AI가 알아서' 칩은 수정에서도 선택 가능 — 저장 시 재추론 요청
- 마감 있는 태스크 → '언젠가' 전환 경로가 생기므로 `noDeadline`이 업데이트 요청에도 필요.
  현재는 마감을 비우면 `scheduleTouched` 재추론이 마감을 되살릴 수 있는데, `noDeadline`
  선언 시 재추론이 마감·시작시간을 채우지 않게 한다.

## 5. 범위 밖

- 브레인 덤프 — 이미 "단서 없으면 deadline null" 규칙 적용됨(변경 없음)
- 대시보드 6버킷·someday 노출 — 변경 없음 ('언젠가' 태스크는 기존 someday 버킷으로 자연 유입)
- 데스크톱(desktop 브랜치)·안드로이드 — 해당 없음

## 6. 검증

- 백엔드: `gradle test`(JDK21 필수) + 신규 테스트
  - n박 일정 등록 성공(시작 오늘·마감 3일 뒤·예상시간 미입력 → 파생 없이 통과)
  - `noDeadline: true` 시 마감·시작시간 추론 미적용(null 확정), 예상시간만 채워짐
  - 부분 입력(예상시간만) 시 마감·시작시간 파생 없음
  - 수정에서 마감 → 언젠가 전환 시 재추론이 마감을 되살리지 않음
- 프론트: `npm run build` + 수동 시나리오
  - 4개 모드 각각으로 생성, 수정 모달 초기 모드 반영·모드 전환, n박 일정 등록
- AI 실응답 검증은 배포 후 스모크 테스트(로컬에 OPENAI_API_KEY 없음) —
  기존 "AI 마감 null" 스모크 항목에 부분 입력 케이스 추가

## 7. 실행 계획 개요

- 구현 순서: ① 백엔드 파생 제거 + `noDeadline` + 프롬프트 정비 + 테스트 →
  ② 생성 폼 모드 칩 → ③ 수정 모달 → ④ 문구·수동 검증
- 상세 단계는 별도 구현 계획 문서(writing-plans)에서 확정
