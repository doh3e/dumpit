# 대시보드 재편성 — 타임테이블 제거 + 할 일/완료 UX 개선 디자인 스펙

- 날짜: 2026-07-11
- 상태: 디자인 4개 섹션 사용자 승인됨, 스펙 검토 대기
- 범위: 사용자 로드맵의 작업 1(타임테이블 제거)과 작업 2(할 일/완료 UX 개선)를 하나의 스펙으로 통합
  (같은 대시보드 화면을 건드리므로 레이아웃 재작업을 한 번으로 끝내기 위함)

## 1. 배경과 목표

대시보드의 원형 타임테이블(`CircularTimetable`)은 시간이 정해진 일만 표시해 활용도가 낮고,
"지금 뭘 해야 하지?"라는 질문에는 그 위의 작은 히어로 카드(nowSuggestion)가 이미 답하고 있다.
할 일 목록은 한 목록에 모든 버킷이 섞여 칩이 난립하고, 완료한 일은 별도 카드로 공간을 차지한다.

**목표**: 타임테이블을 제거하고 "지금 할 일"을 화면의 주인공으로 승격한다. 할 일 목록은
오늘/내일/일주일 탭으로 시점을 하나씩만 보여주고, 완료는 할 일 카드에 접이식으로 통합한다.

## 2. 시간 버킷 — 6버킷 체계 (백엔드 단일 소스)

버킷 계산은 백엔드 `TaskPlanningService` 한 곳에서만 한다. 프론트(대시보드·전체 보기 모달)는
받은 sections를 렌더링만 한다. 이는 추후 안드로이드 앱이 같은 API를 재사용하기 위한 결정이다.

| 버킷 | 기준 (서버 now 기준) |
|---|---|
| `overdue` 마감 지남 | deadline < now |
| `today` 오늘 | deadline ≤ 오늘 23:59:59 |
| `tomorrow` 내일 | deadline ≤ 내일 23:59:59 |
| `next7Days` 일주일 내 | deadline ≤ 7일 후 23:59:59 |
| `later` 그 외 | 그 이후의 deadline (마감 있음) |
| `someday` 언젠가 | deadline 없음 |

기존 `next3Days`(3일 내)는 `tomorrow`로 교체하고, 마감 없는 일은 `later`에서 분리해
`someday`로 독립시킨다. 브레인 덤프로 쌓이는 마감 없는 일이 따로 모여 관리되게 하기 위함이다.
`recentDone`(최근 3일 완료)은 유지한다.

### 백엔드 변경 상세

- `TaskPlanningResponse.TaskPlanningSections` → `overdue, today, tomorrow, next7Days, later, someday, recentDone`
- `TaskPlanningServiceImpl`의 `Bucket` enum과 분류 로직을 위 표대로 변경
- 추천(focusRecommendations) 이유 문구 갱신: "3일 안에 마감돼서…" → "내일 마감이에요" 등
- `timedTasks` 응답 필드 제거 — 소비처가 타임테이블뿐이었음.
  **단, nowSuggestion의 `CURRENT_EVENT` 판정 로직(시간 고정 일정 진행 중 감지)은 서비스 내부에 유지**
- `MAX_RECOMMENDATIONS = 3` 유지 (1위 = 히어로, 2·3위 = 미니 큐)
- 테스트: `TaskPlanningServiceImplTest`를 6버킷으로 갱신하고 경계 케이스 추가
  (오늘 23:59 vs 내일 00:00, 마감 없음 → someday, 7일 초과 → later)

### 호환성

웹 프론트와 동시 배포하므로 구버전 클라이언트 고려는 불필요. 프론트의 planning 실패 fallback은
단순 마감순 목록으로 동작한다(아래 5절).

## 3. "지금 할 일" 히어로 — 상단 전체 폭

`CircularTimetable`을 삭제하고, 기존 히어로 카드를 대시보드 1행 전체 폭으로 확대한다.

- **좌측**: "지금 할 일" 라벨 + 태스크 제목(크게, 갈무리) + 마감 시각 + 추천 이유 메시지
  + **[완료하기] [수정]** 버튼. 완료 시 기존 픽셀 버스트·코인 토스트·로켓 발사 연출 그대로 동작
- **우측**: `OrbitProgress` 진행률 링 (오늘 마감 기준 완료/전체) — 기존 그대로
- **하단**: "다음에 할 일" 미니 큐 — `focusRecommendations`의 2·3위를 작게 표시
  (제목 + 버킷 라벨 + 마감 시각). 클릭 시 수정 모달. 항목이 없으면 섹션 자체를 생략
- 기존 3-상태 유지: ① 오늘 다 비움(🚀) ② 추천 태스크 없음(빈 시간 안내) ③ 추천 태스크 표시.
  ①·②에서는 완료/수정 버튼과 미니 큐를 상황에 맞게 숨긴다 (①은 미니 큐 생략,
  ②는 미니 큐가 있으면 표시)

## 4. "해야 할 일" 카드 — 탭 + 완료 접이식 통합

### 탭

`[오늘] [내일] [일주일] [전부]` — 기본 선택은 오늘.

- **오늘** = `today`, **내일** = `tomorrow`
- **일주일** = `today + tomorrow + next7Days` 합집합 (앞으로 7일 전체를 마감순으로,
  섹션 헤더 없이) — "일주일"의 멘탈 모델은 나머지 5일이 아니라 "앞으로 한 주"이므로
- **전부** = 6버킷 전체를 섹션 헤더(오늘/내일/일주일 내/그 외/언젠가)로 구분한 목록.
  `그 외`·`언젠가` 버킷은 이 탭에서만 노출된다
- **마감 지남**(`overdue`)은 선택된 탭과 무관하게 목록 최상단에 경고색(tone-overdue) 고정
  블록으로 항상 표시

### 항목 row 간소화

체크박스 + 제목 + 마감 시각 + 예상 시간 + 카테고리 칩 + 서브태스크 들여쓰기 표시.
버킷 칩("오늘"/"3일 내")은 탭이 시점을 대신하므로 제거. 상태 칩은 "진행 중"일 때만 표시.
정렬·부모-자식 그룹핑(`groupByParent`)은 백엔드 sections 순서 + 기존 로직 유지.

### 완료 접이식

카드 하단에 `▸ 오늘 완료한 일 (N)` 접이식 섹션 (기본 접힘, N = 오늘 완료 수):

- 펼치면 **오늘 완료한 일만** 표시 (`recentDone`에서 completedAt이 오늘인 항목)
- 각 항목: 취소선 제목 + 완료 시각 + 획득 코인(`calcCompletionCoins` 재계산 — 완료 토스트와
  동일 기준) + 체크 해제 버튼(완료 취소)
- 어제·그제 완료는 전체 보기 모달의 완료 컬럼(최근 3일)에서 확인·체크 해제
- 별도 "완료한 일" 카드는 제거. 장기 완료 기록은 마이페이지 별빛 로그가 계속 담당

### 전체 보기 모달 (TaskBoardModal)

6버킷 + 완료 컬럼으로 갱신: 오늘 / 내일 / 일주일 내 / 그 외 / 언젠가 / 마감 지남 / 완료(최근 3일).
중요도순·마감순 정렬 토글은 유지. 프론트 자체 버킷 계산(`buildSections`)은 제거하고 백엔드
sections만 사용한다.

## 5. 레이아웃과 코드 정리

### 레이아웃 (데스크톱)

```
┌──────────────────────────────┐
│ 지금 할 일 히어로 (전체 폭)      │
└──────────────────────────────┘
┌──────────────┐ ┌─────────────┐
│ 해야 할 일     │ │ 달력         │
│ [탭] 목록      │ │ MiniCalendar│
│ ▸ 오늘 완료    │ │             │
└──────────────┘ └─────────────┘
```

모바일은 기존처럼 세로 1열 (히어로 → 해야 할 일 → 달력).

### 삭제·정리

- `frontend/src/components/CircularTimetable/` 폴더 삭제
- `DashboardPage.jsx`의 로컬 버킷 계산(`getDashboardBucket`, `sortDashboardTasks`,
  `getBucketLabel`, `getFocusRecommendation`) 제거 — planning API 실패 시 fallback은
  마감순 단순 목록으로 축소
- `TaskBoardModal.jsx`의 `buildSections` fallback 제거 (sections 필수화)
- `replacePlanningTask`의 `timedTasks`·`next3Days` 참조를 새 sections 구조로 갱신
- 루틴 시작/끝 설정(localStorage)은 타임테이블 외 사용처(백엔드 추천 등)가 있으므로 유지
